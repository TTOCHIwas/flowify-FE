## 1. 개요

대시보드 페이지는 로그인 후 사용자가 앱 셸에서 가장 먼저 도달하는 홈 화면이다.

이번 작업의 목표는 아래와 같다.

- Figma 시안 기준의 대시보드 레이아웃 구현
- 앱 셸 사이드바의 홈 버튼 진입 경로를 `/dashboard`로 분리
- 현재 존재하는 워크플로우 및 OAuth 토큰 데이터를 활용할 수 있는 영역과, 전용 집계 API가 필요한 영역을 구분
- `docs/CONVENTION.md` 기준에 맞춰 `pages/dashboard` 내부에 화면 전용 구조를 유지
- 랜딩 페이지(`/`)와 앱 내부 홈(`/dashboard`)의 역할을 분리

## 2. 현행 상태

### 2.1 현재 라우팅 구조

현재 라우팅은 아래처럼 구성되어 있다.

- 공개 랜딩 페이지: `/`
- 보호 라우트 앱 셸: `/account`, `/settings`, `/templates`, `/workflows`
- 워크플로우 편집기: `/workflows/:id`

즉 현재 앱 셸 내부에는 별도의 대시보드 페이지가 없고, 사이드바 홈 버튼은 `ROUTE_PATHS.MAIN`을 통해 `/`로 이동한다.

### 2.2 현재 홈 버튼의 문제

현재 구조에서는 앱 셸 내부에서 홈 버튼을 눌렀을 때 공개 랜딩 페이지로 나가게 된다.

이 구조는 아래 이유로 대시보드 역할과 맞지 않는다.

- 앱 내부 홈과 공개 랜딩이 같은 경로를 공유한다.
- 로그인 후 앱 복귀 지점으로 쓰기 어렵다.
- 사이드바의 “홈” 의미가 앱 요약 화면이 아니라 외부 랜딩 페이지가 된다.

따라서 이번 작업에서 홈 버튼의 의미를 “앱 내부 요약 대시보드”로 재정의한다.

### 2.3 현재 재사용 가능한 데이터 소스

현재 프론트에는 아래 데이터 소스가 이미 존재한다.

- 워크플로우 목록: `useInfiniteWorkflowListQuery`
- 워크플로우 목록 단건 페이지 응답 타입: `WorkflowResponse`
- 연결된 서비스 목록: `useOAuthTokensQuery`
- 워크플로우 실행 이력 조회: `useWorkflowExecutionsQuery(workflowId)`

하지만 대시보드 시안에 필요한 “오늘 처리량”, “누적 처리량”, “누적 실행 시간”을 한 번에 제공하는 전용 집계 API는 아직 없다.

## 3. Figma 시안 분석

### 3.1 화면 구조

Figma 노드 `1974:1681`은 앱 셸 내부 콘텐츠 영역에 들어가는 단일 컬럼 대시보드다.

전체 구조는 아래와 같다.

```text
DashboardPage
└── 가운데 정렬된 콘텐츠 래퍼 (폭 1180px)
    ├── 자동화 가동 현황
    │   └── 요약 지표 카드 3개
    ├── 오늘 발생한 에러
    │   └── 자동화 에러 카드 리스트
    ├── 연결된 서비스
    │   └── 인증된 서비스 카드 리스트
    └── 연결 추천 서비스
        └── 미인증 서비스 카드 리스트
```

### 3.2 주요 수치

| 항목 | 값 |
|---|---|
| 콘텐츠 최대 폭 | `1180px` |
| 페이지 내부 패딩 | `48px` |
| 주요 섹션 간 간격 | `48px` |
| 섹션 제목 블록 간격 | `4px` |
| 섹션 제목과 본문 간격 | `24px` |
| 메트릭 카드 높이 | `72px` |
| 메트릭 카드 radius | `8px` |
| 메트릭 카드 가로폭 | `345px` |
| 에러 카드 row padding | `16px` |
| 에러 카드 radius | `10px` |
| 서비스 카드 radius | `10px` |
| 서비스 카드 아이콘 영역 | `38px` |
| 본문 카드 border | `1px solid #f2f2f2` |

### 3.3 시안 의도 해석

시안은 대시보드를 “운영 상태를 빠르게 훑는 화면”으로 설계한다.

- 상단 3개 지표 카드로 전체 현황을 한 번에 파악한다.
- 에러 섹션은 문제가 있는 자동화를 리스트로 빠르게 확인하게 한다.
- 연결된 서비스와 연결 추천 서비스를 분리해 현재 상태와 다음 액션을 같이 보여준다.
- 전체 구조는 카드 갤러리보다 “운영 대시보드”에 가까운 정보 밀도를 가진다.

## 4. 구현 범위

### 4.1 포함 범위

- `/dashboard` 페이지 라우트 추가
- 앱 셸 사이드바 홈 버튼을 `/dashboard`로 연결
- Figma 기준의 단일 컬럼 대시보드 레이아웃 구현
- 상단 요약 지표 카드 3개 UI 구현
- 에러 자동화 카드 섹션 UI 구현
- 연결된 서비스 / 연결 추천 서비스 카드 섹션 UI 구현
- 현재 API로 가능한 영역은 실제 데이터 연결
- 현재 API로 불가능한 지표는 명시적인 placeholder 정책 적용

### 4.2 제외 범위

- 대시보드 전용 집계 API 신규 설계 또는 백엔드 구현
- 서비스 연결/해제 액션 버튼 추가
- 에러 카드에서 상세 모달 또는 상세 화면 진입
- 실시간 polling 기반 대시보드 자동 갱신
- 지표 카드 클릭 후 drill-down 화면 제공

## 5. 라우팅 및 내비게이션 설계

### 5.1 경로 정의

이번 작업에서 새 상수 경로를 추가한다.

```ts
ROUTE_PATHS.DASHBOARD = "/dashboard"
```

### 5.2 라우트 배치

`/dashboard`는 로그인 후 접근 가능한 앱 내부 홈이므로 `ProtectedRoute + AppShellLayout` 아래에 둔다.

정리 기준:

- `/`는 공개 랜딩 페이지로 유지
- `/dashboard`는 보호 라우트 대시보드 페이지로 추가
- 사이드바 `home` 아이템은 `ROUTE_PATHS.DASHBOARD`를 가리키도록 변경

### 5.3 홈 버튼 의미 재정의

이번 작업 이후 홈 버튼 의미는 아래처럼 바뀐다.

| 항목 | 기존 | 변경 후 |
|---|---|---|
| 홈 버튼 대상 | `/` | `/dashboard` |
| 화면 성격 | 공개 랜딩 | 앱 내부 대시보드 |
| 레이아웃 | `LandingLayout` | `AppShellLayout` |

이로써 랜딩 페이지와 앱 내부 홈의 책임을 분리한다.

## 6. UI 설계

### 6.1 페이지 레이아웃

페이지는 기존 `TemplatesPage`, `WorkflowsPage`와 같은 패턴으로 얇게 유지한다.

기준:

- 페이지 래퍼: `w="full"`, `maxW="1180px"`, `mx="auto"`
- 상위 정렬: `VStack align="stretch"`
- 섹션 간 gap: `12`(`48px`)

`AppShellLayout`이 이미 앱 셸 공통 padding을 제공하므로, 대시보드는 내부 콘텐츠 래퍼의 폭과 섹션 간격만 책임진다.

### 6.2 섹션 구조

대시보드는 아래 4개 섹션으로 구성한다.

1. `DashboardOverviewSection`
2. `DashboardErrorSection`
3. `ConnectedServicesSection`
4. `RecommendedServicesSection`

각 섹션은 제목과 설명을 포함한 공통 heading block을 가진다.

### 6.3 상단 요약 지표 카드

상단 제목 영역 카피는 시안 기준으로 아래를 사용한다.

| 항목 | 텍스트 |
|---|---|
| 제목 | `자동화 가동 현황` |
| 부제 | `내가 구축한 자동화 시스템 목록` |

지표 카드는 3개다.

| 카드 | 라벨 | 값 |
|---|---|---|
| 카드 1 | `오늘 처리량` | `000` 또는 집계 가능 시 실제 값 |
| 카드 2 | `누적 처리량` | `000` 또는 집계 가능 시 실제 값 |
| 카드 3 | `누적 실행 시간` | `00:00:00` 또는 집계 가능 시 실제 값 |

스타일 기준:

- 높이 `72px`
- padding `16px`
- gap `24px`
- border `1px solid`
- `borderColor="border.default"`
- `borderRadius="8px"`
- 배경 `bg="bg.surface"`

### 6.4 오늘 발생한 에러 섹션

섹션 카피:

| 항목 | 텍스트 |
|---|---|
| 제목 | `오늘 발생한 에러` |
| 부제 | `오늘 발생한 에러` |

에러 카드는 아래 정보를 가진다.

- 시작 서비스 배지
- 연결 화살표
- 종료 서비스 배지
- 자동화 이름
- 메타 정보(`00전 변경됨 | 00/00 구축` 형태)
- 상태 텍스트(`중지됨` 등) 또는 상태 아이콘
- 하위 에러 메시지 row 리스트

하위 에러 row 구성 기준:

- 각 row는 에러 메시지와 함께 관련 노드/서비스 아이콘을 노출한다.
- 가능하면 warning의 `nodeId`로 실제 워크플로우 노드를 찾아 아이콘을 결정한다.
- `nodeId`로 아이콘을 찾지 못하면 `sourceType`, `targetType` 기반 fallback 규칙을 사용한다.
- 단순 경고 아이콘만 고정으로 쓰지 않는다.

상호작용 기준:

- 기본 상태에서는 자동화 헤더 row만 노출한다.
- 카드 클릭 시 하위 에러 메시지 row가 아래로 펼쳐진다.
- 다시 클릭하면 접힌다.
- 상태 표현은 워크플로우 목록과 같은 어휘를 우선 참고한다.
- 필요하면 `재생`, `중지` 아이콘을 함께 사용한다.

스타일 기준:

- 카드 padding `16px`
- radius `10px`
- border `1px solid #f2f2f2`
- 에러 상태 강조용 연한 red shadow 허용
- 하위 에러 row는 내부 border box 구조

### 6.5 연결된 서비스 섹션

섹션 카피:

| 항목 | 텍스트 |
|---|---|
| 제목 | `연결된 서비스` |
| 부제 | `인증된 서비스 관리` |

서비스 카드 정보:

- 대표 아이콘
- 서비스 이름
- 상태 문구

상태 문구 기준:

- 연결됨: `n분 전 인증됨` 또는 fallback 문구
- 추가 정보가 없으면 단순 `연결됨`

### 6.6 연결 추천 서비스 섹션

섹션 제목은 `연결 추천 서비스`를 사용한다.

카드 구조는 연결된 서비스 카드와 동일하되, 상태 문구는 `인증 전`으로 고정한다.

추천 서비스는 1차 구현에서 아래 우선순위로 산출한다.

1. 워크플로우/템플릿에서 자주 쓰는 서비스
2. 현재 미연결 서비스
3. 시안 유지용 fallback 서비스 목록

### 6.7 반응형 기준

모바일 시안은 없으므로 아래 원칙으로 축소한다.

- 메트릭 카드는 `base`에서 세로 스택
- 서비스 카드는 wrap 유지
- 에러 카드 내부 헤더는 좁은 폭에서 세로 정렬 허용
- 메타 텍스트는 2줄까지 허용

## 7. 데이터 및 API 정합성

### 7.1 현재 API로 바로 연결 가능한 영역

| 영역 | 데이터 소스 | 1차 구현 가능 여부 |
|---|---|---|
| 연결된 서비스 | `useOAuthTokensQuery()` | 가능 |
| 추천 서비스 | 연결된 서비스 목록 + 프론트 fallback 규칙 | 가능 |
| 자동화 제목/서비스 아이콘 | `useInfiniteWorkflowListQuery()` | 가능 |
| 자동화 상태 | `workflow.active` | 가능 |

### 7.2 현재 API로 직접 집계할 수 없는 영역

| 영역 | 부족한 점 |
|---|---|
| 오늘 처리량 | 전역 당일 실행 집계를 효율적으로 제공하는 API 없음 |
| 누적 처리량 | 전역 누적 실행 집계를 효율적으로 제공하는 API 없음 |
| 누적 실행 시간 | 전역 실행 시간 합계 API 없음 |
| 오늘 발생한 에러 | 날짜 기준 실패 실행 집계 API 없음 |

즉 시안의 핵심 운영 지표는 현재 API만으로 정확하게 재현할 수 없다.

### 7.3 1차 구현 데이터 정책

이번 1차 구현에서는 아래 정책을 따른다.

#### 지표 카드

요구사항 기준 정의:

- `오늘 처리량`: 오늘 내 워크플로우 실행 횟수
- `누적 처리량`: 전체 내 워크플로우 실행 횟수
- `누적 실행 시간`: 전체 실행 시간 합계

현재 판단:

- 실행 이력 API는 `GET /workflows/:id/executions`라 워크플로우별 조회만 가능하다.
- 기술적으로는 모든 워크플로우에 대해 실행 이력을 순회 조회한 뒤 프론트에서 합산할 수 있다.
- 하지만 이 방식은 대시보드 진입 시 요청 수가 워크플로우 개수에 비례해 증가한다.
- 따라서 지표 3개는 전용 집계 API 또는 글로벌 execution summary API를 권장한다.

- 현재 API만으로 계산할 수 없는 값은 시안과 같은 placeholder 형식으로 노출
- placeholder는 `000`, `00:00:00`처럼 명시적으로 스텁임이 드러나는 값 사용
- 임의 추정치나 의미가 다른 값을 끼워 맞춰 사용하지 않는다

#### 에러 섹션

- 요구사항 기준으로 이 섹션은 실행 로그 기반 에러 조회를 우선 검토한다.
- 현재 실행 상세 타입에는 `ExecutionDetail.nodeLogs[].error` 구조가 있어, 실제 에러 메시지 추출은 가능하다.
- 다만 대시보드에서 필요한 것은 “오늘 발생한 에러”이므로, 날짜 기준으로 전체 워크플로우 실행 이력을 모아야 한다.
- 현재 API 구조에서는 워크플로우별 실행 목록을 개별 조회해야 하므로, 정확한 today error 집계는 전용 집계 API가 더 적합하다.
- 1차 구현에서는 구조와 펼침 인터랙션을 우선 맞추고, 실제 today 집계 연결은 후속 정합화 대상으로 둔다.
- 현재 워크플로우 경고(`ValidationWarning`)에는 `nodeId`, `sourceType`, `targetType`, `message`가 있으므로, 펼친 row의 아이콘은 프론트에서 바로 계산 가능하다.
- 즉 “펼친 에러 row에 노드 아이콘을 보여주는 것” 자체는 API 변경 없이 프론트 수정으로 처리한다.

#### 연결된 서비스

- 연결된 서비스 목록 자체는 현재 `GET /oauth-tokens`로 조회 가능하다.
- 따라서 “목록을 보여주기 위한 신규 API”는 필수가 아니다.
- `OAuthTokenSummary`의 `service`, `connected`, `expiresAt`를 활용한다.
- 다만 시안의 `00전 인증됨`처럼 정확한 인증 시점을 보여주려면 `connectedAt` 또는 이에 준하는 timestamp 필드가 추가로 필요하다.
- `expiresAt`는 인증 시점이 아니라 만료 시점이므로, 이를 그대로 `n분 전 인증됨` 계산에 사용하지 않는다.
- `accountEmail`은 시안에 없으므로 1차 구현에서 직접 노출하지 않는다.

#### 추천 서비스

- 연결되지 않은 주요 서비스 키 목록을 프론트에서 계산
- 현재 서비스 아이콘 시스템에서 렌더 가능한 서비스만 1차 후보로 사용

### 7.4 향후 권장 API

대시보드 품질을 올리려면 전용 요약 API가 필요하다.

권장 예시:

```json
{
  "todayProcessedCount": 12,
  "totalProcessedCount": 483,
  "totalExecutionDurationSeconds": 9372,
  "todayErrors": [
    {
      "workflowId": "wf_1",
      "workflowName": "자동화 제목",
      "startService": "calendar",
      "endService": "notion",
      "status": "stopped",
      "messages": ["에러 내용"]
    }
  ]
}
```

이번 작업에서는 이 API를 전제로 구현하지 않는다.

연결 서비스 영역에서 시안 정합도를 높이려면 아래와 같은 응답 확장도 유용하다.

```json
{
  "service": "notion",
  "connected": true,
  "connectedAt": "2026-04-14T09:30:00Z",
  "accountEmail": "user@example.com"
}
```

## 9. 구현 단계 및 커밋 계획

### 9.1 권장 단계

| 단계 | 구현 범위 | 완료 기준 | 권장 커밋 메시지 |
|---|---|---|---|
| 1 | 문서 + 라우트 구조 | `/dashboard` 경로와 홈 버튼 방향이 문서/코드 기준으로 정리됨 | `docs: 대시보드 페이지 설계 기준을 정리` |
| 2 | 페이지 뼈대 + 섹션 구조 | `pages/dashboard` 구조가 생기고 Figma 레이아웃 틀이 보임 | `feat: 대시보드 페이지 기본 레이아웃을 추가` |
| 3 | 연결 서비스/추천 서비스 | 실제 OAuth 토큰 기반 카드가 렌더링됨 | `feat: 대시보드 서비스 연결 섹션을 구현` |
| 4 | 에러 카드 구조 | 워크플로우 기반 자동화 카드가 에러 섹션에 렌더링됨 | `feat: 대시보드 에러 카드 섹션을 구현` |
| 5 | 지표 카드/마감 | 상단 지표 카드와 반응형/상태 표현이 마감됨 | `refactor: 대시보드 지표 카드와 상태 표현을 다듬음` |

### 9.2 구현 원칙

- 문서와 코드의 라우팅 의도는 반드시 같이 맞춘다.
- `widgets`로 올리지 않고 `pages/dashboard` 내부에 유지한다.
- `useDashboardData`는 query 조합과 파생 상태만 담당한다.
- 카드 렌더링 컴포넌트는 query를 직접 호출하지 않는다.
- API로 계산 불가능한 값은 placeholder로 고정하고, 의미가 다른 데이터로 대체하지 않는다.
- 요구사항이 바뀌면 문서를 먼저 갱신한 뒤 코드를 수정한다.

## 11. 후속 작업

### 11.1 전용 대시보드 집계 API

- 오늘 처리량
- 누적 처리량
- 누적 실행 시간
- 날짜 기준 실패 자동화 목록

이 4가지는 전용 API가 생겨야 시안 의미를 정확히 충족할 수 있다.

### 11.2 에러 모델 정교화

현재 워크플로우 `warnings`는 구성 경고에 가깝다.

후속 과제:

- 실행 실패와 구성 경고를 분리
- 날짜 기준 실패 실행만 대시보드에 노출
- 에러 카드 클릭 시 펼침/접힘 상태와 상세 실행 이력 연결을 정교화
- 상태 텍스트와 재생/중지 아이콘 표현을 워크플로우 목록과 일관되게 정리

### 11.3 서비스 추천 규칙 고도화

- 사용자별 자주 쓰는 서비스 기반 추천
- 템플릿/워크플로우 구성 기반 추천
- 연결 액션 진입 버튼 추가

### 11.4 연결 서비스 응답 확장

- `connectedAt` 추가
- 서비스별 표시용 메타 정보 정리
- 대시보드와 계정 페이지가 같은 응답 스펙을 공유하도록 정리

## 결론

이번 대시보드 작업의 핵심은 “홈 버튼을 앱 내부 요약 화면으로 재정의”하는 것이다.

구현 우선순위는 아래와 같다.

1. `/dashboard` 라우트와 사이드바 홈 버튼 연결
2. Figma 기준의 섹션/카드 레이아웃 구현
3. 현재 API로 연결 가능한 영역은 실제 데이터 사용
4. 현재 API로 불가능한 집계 값은 placeholder 정책으로 명확히 처리

이 문서는 `/dashboard` 페이지 구현의 기준 문서로 사용한다.
